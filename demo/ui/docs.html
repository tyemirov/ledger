Ledger Integration Guide

How to run the ledger as a standalone gRPC service (`ledgerd`) or embed the Go library (`pkg/ledger`).

1. Running the gRPC microservice

  1. Build the binary: go build ./cmd/credit
  2. Provide a database via DATABASE_URL and optionally GRPC_LISTEN_ADDR (defaults to :50051). 
DATABASE_URL=sqlite:///tmp/ledger.db GRPC_LISTEN_ADDR=:50051 ./ledgerd

SQLite files are created automatically. For Postgres, ensure the database exists and the configured user can create tables and indexes. The service applies its schema on startup via GORM.

2. Embedding the Go library

Use pkg/ledger in-process and wire a store implementation (GORM or pgx) to the service. 

import (
    "github.com/MarkoPoloResearchLab/ledger/pkg/ledger"
    "github.com/MarkoPoloResearchLab/ledger/internal/store/gormstore"
)

func newLedgerService(db *gorm.DB, clock func() int64) (*ledger.Service, error) {
    store := gormstore.New(db)
    return ledger.NewService(store, clock)
}

Validation happens at the edge: construct TenantID, UserID, LedgerID, PositiveAmountCents, IdempotencyKey, and MetadataJSON before calling the service.

3. Error contracts

The ledger returns sentinel errors you can match with errors.Is.

  • Input validation: ErrInvalidTenantID, ErrInvalidUserID, ErrInvalidLedgerID, ErrInvalidReservationID, ErrInvalidIdempotencyKey, ErrInvalidAmountCents, ErrInvalidMetadataJSON
  • Business rules: ErrInsufficientFunds, ErrDuplicateIdempotencyKey
  • Reservation state: ErrReservationExists, ErrUnknownReservation, ErrReservationClosed

Store adapters wrap errors with ledger.OperationError so downstream callers can log stable codes (operation.subject.code).

4. Choosing an approach

  • Multiple services / polyglot clients: deploy ledgerd and call it via gRPC.
  • Single Go service needs credits: embed pkg/ledger + gormstore/pgstore.
  • Need both: host ledgerd for general use; embed the library where tighter coupling helps.
